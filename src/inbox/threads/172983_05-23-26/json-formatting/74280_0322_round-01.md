# Clarifying round 1 — `json-formatting`

Source paragraph (full informal spec):

> - All JSON that either agents or automated scripts write to output files,
>   terminal/CLI, or agent chat window must be pretty-formatted: indent=2,
>   one key/value per line.
> - Any commit hash values (e.g. `contentHash`) must be the abbreviated hash,
>   NOT the full hash.
> - Existing JSON must be migrated to the new convention.

The following questions must be answered before the canonical spec can be
authored. Each question cites the source clause it targets.

---

- **Q1 — Hash abbreviation length** [scope, constraints]
  The spec says commit hash values such as `contentHash` MUST use the
  "abbreviated hash, NOT the full hash," but does not define the target
  length. What character count defines "abbreviated"?
  Options: (a) 7-character `git rev-parse --short HEAD` default,
  (b) 12-character, (c) match whatever `git rev-parse --short` returns
  for the repo at the time of writing, (d) a fixed length you specify.
  — *Source: `src/inbox/in/json_formatting.md` line 2,
  `{kind: 'lines', path: 'src/inbox/in/json_formatting.md', range: [2, 2], contentHash: ebfc73c552f1feeeda24951ad0808b63f47e07956e36e4119862bd6c076a7955}`*

- **Q2 — Does "abbreviated hash" apply to `contentHash` citation fields?**
  [scope]
  The glossary (`src/memory/handbook/glossary.md` line 204) defines
  `contentHash` as "the SHA-256 of the cited file at citation time"—a
  full 64-character hex digest, not a git commit hash.
  Does the spec's abbreviation rule apply only to **git commit hashes**
  (e.g., `Bootstrap-Phase` trailer values, run-log commit refs), or also
  to SHA-256 content hashes in dual-anchor citation fields?
  — *Source: `src/inbox/in/json_formatting.md` line 2,

  "Output files" is undefined. Does the pretty-print rule apply to:

  (including `src/memory/features/*/index.json`,
  `src/work/*/policy-compliance.json`, compliance fixtures under
  `tests/compliance/`, and `package.json` files), or
  (b) only files within explicitly designated agent/script output paths
  (e.g., `src/inbox/`, `src/work/`, `src/memory/`), or
  (c) a specific enumerated list of paths you define?
